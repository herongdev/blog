在 MyBatis 的 Mapper 中，你可以直接使用对象的子属性。假设你有一个 PlaceBaseVO 对象并且该对象包含一个 Address 对象，那么你在 MyBatis 的 Mapper 中可以像这样使用它：

xml

\<!-- 插入新的ActivityPlaces，并获取自动生成的ID --\>
\<insert id="add" useGeneratedKeys="true" keyProperty="id"\>
INSERT INTO liuwa.activity_places (name_en, name_zh, address_id, /* ... other fields ... */)
VALUES (#{nameEn}, #{nameZh}, #{addressId}, /* ... other fields ... */)
\</insert\>

\<!-- 插入新的Address，并获取自动生成的ID --\>
\<insert id="addAddress" useGeneratedKeys="true" keyProperty="address.id"\>
INSERT INTO liuwa.activity_place_address (province, city, /* ... other fields ... */)
VALUES (#{address.province}, #{address.city}, /* ... other fields ... */)
\</insert\>
这里，#{nameEn} 和 #{nameZh} 是 PlaceBaseVO 对象的顶级属性，而 #{address.province} 和 #{address.city} 是 Address 子对象的属性。

如何处理多余的 address 参数
对于这种场景，通常有两种做法：

服务层处理: 在服务层先保存 Address 对象，获取生成的 addressId，然后再用这个 addressId 和其他用户输入的信息来保存 Place 对象。这样，当调用地点表（Place）的 Mapper 方法时，你不会传入多余的 address 信息。

自定义SQL参数: 另一个方法是在 Mapper 中使用 \<if\> 标签来有选择地插入字段。这样，你可以根据字段是否为空或是否已设置来动态生成SQL。

第一种方法通常更简单，更易于维护，也更符合单一职责原则。

下面是一个伪代码示例：

java
Copy code
public class PlaceService {

@Autowired
private PlaceMapper placeMapper;

public Long createPlace(PlaceBaseVO placeBaseVO) {
// Step 1: Insert address and get generated address ID
Long addressId = placeMapper.addAddress(placeBaseVO.getAddress());   // Step 2: Set address ID to placeBaseVO
placeBaseVO.setAddressId(addressId);   // Step 3: Insert place
return placeMapper.add(placeBaseVO);
}
}
这样，你可以确保 placeMapper.add() 方法只会接受和处理它需要的信息。

确实，placeBaseVO 里还包含了 address 的信息，但如果你的 Mapper 方法和对应的 SQL 查询没有明确地使用这个 address 字段，那么这个额外的信息就会被忽略。

如果你在 placeMapper.add() 方法中的 SQL 查询只指定了你实际需要插入到 liuwa.activity_places 表中的字段（并且没有包括 address 字段或其子字段），那么 address 信息就不会被使用。

例如，假设你的 Mapper XML 文件中的插入语句如下：

xml
Copy code
\<!-- 插入新的ActivityPlaces，并获取自动生成的ID --\>
\<insert id="add" useGeneratedKeys="true" keyProperty="id"\>
INSERT INTO liuwa.activity_places (name_en, name_zh, address_id, /* ... other fields ... */)
VALUES (#{nameEn}, #{nameZh}, #{addressId}, /* ... other fields ... */)
\</insert\>
这里只用到了 nameEn, nameZh, 和 addressId 字段，没有用到 address，所以即便 placeBaseVO 对象里有 address 的信息，这个信息也不会影响这个 SQL 查询。

所以，简而言之，没有关系——你不用它就行了。当然，如果你觉得这样做在逻辑上不够清晰，你也可以创建一个新的 VO 对象，只包含用于插入 liuwa.activity_places 表的字段，而不包括 address 信息。这样做更清晰，但会增加代码的复杂性。

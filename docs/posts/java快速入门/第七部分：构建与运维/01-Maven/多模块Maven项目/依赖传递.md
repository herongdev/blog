首先，让我们来明确一些事情：

你的pom.xml文件中确实没有明确包含名为system的模块依赖。
你的代码示例中直接引用了com.ms.system.repository.FileRepository，这是system模块中的一个类。
考虑以下可能性：

- 传递依赖: 如果system模块是其他你已经包含的模块（例如ms-framework、ms-quartz、ms-generator等）的依赖，那么system模块中的类将被传递到当前模块中。这就是为什么你可以直接引用它而不需要明确地在pom.xml中包含system模块。
- 同一工作空间的其他模块: 如果你的整个项目是一个多模块Maven项目，那么IDE（例如IntelliJ IDEA或Eclipse）可能会允许你在一个模块中引用另一个模块的代码，即使它们在Maven依赖关系中并没有明确连接。这主要是为了在开发期间提供方便，但是当你尝试构建或运行项目时，如果没有正确的Maven依赖，你可能会遇到问题。
- 本地的未打包类: 如果com.ms.system.repository.FileRepository类是在本地项目中但未在Maven中被管理，并且和当前模块在同一源代码目录树下，那么你可能可以在IDE中直接引用它，但这在真正的Maven构建中可能会失败。

建议的解决方案：

- 检查传递依赖: 使用mvn dependency:tree命令查看当前模块的所有依赖，包括传递依赖。这将帮助你确定system模块是否作为传递依赖被包含进来。
- 明确依赖: 如果你的代码需要system模块，那么在pom.xml中明确地添加该模块作为依赖是一个好习惯。这样可以确保无论在哪里构建，项目都能正确地构建和运行。
- 在IDE中检查配置: 检查你的IDE配置和Maven模块设置，确保它们之间的同步，以防止此类差异。

结论：即使在开发时没有问题，也需要确保Maven的依赖关系完整和正确，以保证在不同的环境和构建过程中都能稳定工作。

从给出的项目结构来看，你的项目是一个基于Maven的多模块Java项目。我们可以采用异步转换的策略来实现文件上传后的转换功能。以下是具体的实施步骤：

1. 添加必要的依赖
在项目的父pom.xml或具体需要用到的模块（例如ms-admin）的pom.xml中添加Apache POI和PDF相关的依赖。

2. 设计数据库结构
在数据库中设计一个表，记录每个上传文件的信息和转换状态。字段可以包括：

- 文件ID
- 原始文件路径
- 转换后的文件路径
- 文件类型（doc, docx, xls, xlsx, ppt, pptx, pdf）
- 转换状态（待转换，转换中，转换完成，转换失败）
- 上传时间
- 转换完成时间等

3. 文件上传逻辑

在com.ms.web.controller.common包中：

- 当用户上传文件，保存文件到服务器或云存储，并将文件记录到数据库，状态设置为“待转换”。
- 你可以使用消息队列（如果项目已集成）或其他方式将待转换的文件ID发送到后台转换服务。

4. 实现文件转换服务
选择一个模块，例如ms-system，实现文件转换逻辑：

- 监听消息队列或其他方式获取待转换的文件ID。
- 根据ID从数据库中获取文件的详细信息。
- 使用Apache POI和PDF库进行文件转换。
- 转换完成后，更新数据库中的文件记录，设置转换状态为“完成”，并记录转换后的文件路径。

5. 文件预览逻辑
在com.ms.web.controller.common包中：

- 当用户请求预览文件，根据文件ID从数据库中查询文件信息。
- 如果文件已转换，直接返回转换后的文件供用户预览。
- 如果文件还在转换队列中或转换失败，返回相应的提示信息给用户。

6. 异常处理和资源清理

- 如果转换过程中发生异常，捕获异常并记录日志，更新数据库中的文件记录为“转换失败”。
- 定期清理长时间未被访问或预览的转换文件，以释放存储空间。

7. 安全性和优化

- 考虑文件的存储安全性，避免文件被外部非法访问。
- 如果发现转换服务经常有大量的文件在队列中等待，考虑扩展转换服务，例如增加转换服务的实例数量或使用更强大的服务器。
- 根据业务情况，对文件大小、类型等进行限制，避免恶意文件上传。

这只是一个大致的实施方案，根据项目的实际情况和需求，可能需要进一步细化和调整。

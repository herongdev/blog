# 为什么要重新系统学习 Java？

你可能已经看过很多 Java 语法点，也能在 AI 帮助下生成代码，但一进真实项目还是会卡在这些地方：不知道类之间怎么协作，看不懂 Spring 注解背后的边界，遇到堆栈只能复制报错搜索，改接口时不确定 Controller、Service、Mapper 应该各自写什么。

重新系统学 Java，不是为了从零背语法，而是为了补上“能读、能改、能解释、能验证”的后端能力。

## 学完后你应该能做什么

| 工作场景 | 你要具备的能力 | 本系列对应证据 |
|----------|----------------|----------------|
| 接手一个 Java 项目 | 识别 JDK、Maven、包结构、启动类、配置文件和依赖来源 | 环境记录、`pom.xml` 说明、项目结构图 |
| 修改一个接口 | 说明 Controller、Service、Mapper、DO/DTO/VO 的职责 | 任务 API 练习、接口测试、调用链说明 |
| 定位一次报错 | 读堆栈、找异常类型、确认输入和配置问题 | 错误复盘、失败用例、修复记录 |
| 写一个小功能 | 用集合、异常、时间、文件、数据库和测试完成闭环 | 可运行代码、`mvn test` 输出、验收清单 |
| 复盘面试题 | 不只背答案，还能拿自己的练习解释设计取舍 | 学习日志、能力地图、Demo 用例 |

这套资料能帮你形成项目证据和表达能力，但不承诺直接带来工作机会。你真正能带走的是一套可迁移的学习方法：遇到新代码，先判断它解决什么问题，再看它放在哪一层，最后用运行结果验证理解。

## 为什么 Java 仍值得系统学

| 领域 | Java 的位置 |
|------|-------------|
| 企业后端 | Spring Boot、微服务、批处理、内部平台和存量系统仍大量使用 Java |
| Android 与 JVM 生态 | Kotlin 很常见，但 Android SDK、旧项目、JVM 概念和 Java 互操作仍绕不开 |
| 中间件与基础设施 | Kafka、Elasticsearch、Netty、Hadoop 等生态经常需要读 Java 示例或源码 |
| 维护与协作 | 真实工作常常不是新建项目，而是读旧代码、修 bug、补测试、做迁移 |

如果你会 Kotlin、JavaScript 或其他语言，Java 仍然值得补一遍，因为它训练的是 JVM 后端项目的共同语言：类型、对象、集合、异常、并发、构建、配置和分层。

## 本系列怎么学

第一轮只追求一件事：做出能运行、能测试、能讲清楚的小项目。

1. 从 [Java 快速入门教程](./README.md) 进入。
2. 按 [01-课程路线图](./01-课程路线图.md) 完成环境、语法、对象、核心 API、Maven、Spring Boot、数据库和质量补齐。
3. 用 [02-当前实践说明](./02-当前实践说明.md) 判断哪些旧文章适合照学，哪些只适合当历史项目材料。
4. 用 [03-学习验收清单](./03-学习验收清单.md) 记录每个阶段的运行结果、失败用例和错误复盘。
5. 用 [0-提示词](./0-提示词.md) 把新遇到的 Java 代码继续归类到这套知识库。

## 学习原则

- 先能跑，再求深：每个阶段至少留下一个可执行命令、接口请求或测试结果。
- API 不靠背：重点判断什么时候用 `List`、`Map`、`Stream`、`Optional`、文件、时间和异常处理。
- 面向对象不要跳过：能解释类、接口和对象协作，后面的 Spring 才不会变成背注解。
- 数据库和接口要连起来：请求参数进入 Controller 后，必须能追到 Service、Mapper、SQL 和返回对象。
- 错误要写下来：每次报错都记录输入、预期、实际、定位过程和修复方式。
